PART 1
동기와 문제 정의
왜 "에이전트의 인프라 비용"을 측정해야 하는가

왜 이 논문인가

LLM의 발전 패러다임이 "모델·데이터 키우기"에서 "추론 시 더 깊이 생각하기 (test-time scaling)"로 이동했고, 그 정점에 있는 AI 에이전트는 plan → tool → observe → refine 을 반복.

이미 알려진 것

  • 정적 LLM의 serving 비용은 잘 측정됨 (vLLM, Orca 등)
  • ChatGPT 1쿼리 ≈ 웹 검색 10배 전력
  • xAI Colossus = H100 100K, 150 MW

아직 측정 안 된 것 (=논문의 갭)

  • 5종 에이전트의 단대단 인프라 비용
  • LLM 호출 / 도구 호출 / GPU idle 분해
  • Test-time scaling의 한계 비용
  • 데이터센터 단위 전력 함의
핵심 질문 — "ChatGPT처럼 한 번에 답하는 LLM 대비, Plan하고 Tool 부르고 Reflect하는 AI 에이전트는 인프라 측면에서 얼마나 더 비쌀까? 그리고 그만큼 정확도가 오를까?"

Fig 1 — 정적 vs 동적 추론 논문 only

해설
  • (a) 추론 능력 없는 기본 LLM: 입력 → 단일 forward pass → 출력. 예: "Hello"를 번역.
  • (b) 정적 추론 LLM (CoT 등): 내부적으로 중간 단계 생성. 모든 reasoning이 LLM 안에서 일어남.
  • (c) 동적 추론 AI 에이전트: plan → 외부 도구 호출 → 결과 관찰 → refine. 컨텍스트가 누적되며 회색/분홍/청록/녹색/노랑 5색 토큰 모두 등장.

Fig 2 — AI Agent의 5요소 구조 논문 only

해설
  • ① Agent core — Actor / Planner / Reflection module 등 LLM 역할 분배.
  • ② Memory — 단기(대화 trace) + 장기(사용자 선호, 과거 시도 결과).
  • ③ Plan — 시퀀스 또는 DAG로 하위 작업 조직.
  • ④ Tools — 외부 환경과 상호작용. 구조화 명령 → 호출 → 결과 컨텍스트 통합.
  • ⑤ Workflow — 위 4요소가 반복 상호작용하는 패턴. ReAct/Reflexion/LATS/LLMCompiler가 서로 다른 패턴.

Fig 3 — 5종 에이전트의 실행 타임라인 논문 only

해설
  • CoT: LLM 호출 1회로 끝. 도구 사용 없음.
  • ReAct: Thought → Action(Tool) → Observation 반복. 가장 직관적인 에이전트 패턴.
  • Reflexion: ReAct의 trace를 끝낸 뒤 별도 "reflection" 단계가 평가 → 다음 trial.
  • LATS: 트리 검색 (MCTS). 각 노드 확장마다 자식 후보를 여러 개 sampling — LLM 호출이 자식 수만큼 폭증.
  • LLMCompiler: 먼저 DAG 형태의 plan을 생성 → 독립 노드를 비동기 병렬 실행.

측정 결과 — 우리가 재현한 핵심 figure
좌:논문 · 우:우리 재현 · 하단:비교 해설

Fig 4 — 요청당 LLM·Tool 호출 횟수

해설 (Tool 카운트 사후 복구 반영)
  • 일치 패턴: LATS가 다른 에이전트 대비 압도적으로 많은 LLM 호출. 논문 71회/query, 우리 193회/query — 같은 방향, 트리 폭 차이로 더 큼.
  • Tool 카운트 측정 한계 (L4): ReAct(8.5)만 LangChain 표준 콜백으로 직접 측정. Reflexion/LATS/LLMCompiler는 자체 wrapper로 콜백 우회 → 처음에는 0으로 나옴.
  • 사후 복구 (이번 업데이트): 구조적 모델로 재추정 — Reflexion ≈ 14 (n_llm − 2×reflections − 1), LATS ≈ 55 (n_tree_expansions × 5 children × 3 actions). 모두 ReAct 8.5 보다 많음 = 도구를 더 자주 부른다는 패턴 일치.
  • LLMCompiler 예외: n_llm = 1.18 로 plan 호출만 발생 + 최종 답변이 50쿼리 중 41개에서 빈 문자열 → executor 자체가 작동 안 함. Tool 카운트 복구 불가능 ("?"로 표시).

Fig 5 — 지연 시간 분해 (Latency Breakdown)

해설 (Tool latency 사후 추정 반영)
  • ReAct (측정 신뢰): Prefill 6.6s (16%) · Decode 14.7s (36%) · Tool wait 17.7s (44%) — HotpotQA에서 Wikipedia tool wait가 단일 최대 컴포넌트, 논문과 일치.
  • Reflexion/LATS (측정 한계 + 사후 추정): tool_ms = 0 (L4 콜백 우회) → recovered count × ReAct rate(2.08s/call)로 빗금 막대 추정. Reflexion ≈ 29s (33% of 87s), LATS ≈ 114s (11% of 1037s).
  • Raw sum > 100% 경고: Reflexion 157%, LATS 328% — 100ms 폴링 thread가 동시 실행 구간에서 prefill·decode를 중복 attribution. 즉 prefill/decode 막대 자체도 실제보다 부풀려진 상태 → 100%로 정규화 표시.
  • LLMCompiler 회색 overhead 84%: tool 호출이 "overhead" 버킷에 들어감 (executor 깨짐). 실제로 Wikipedia 호출 중 대기 시간이지만 분류 불가.

Fig 6 — GPU 활용률 (가장 강한 재현 검증)

해설 — 본 재현의 가장 강력한 검증 포인트
  • 논문 §IV-A.2: ReAct HotpotQA GPU idle = 54.5% (NVIDIA DCGM) → 우리 측정 54.98% (wall-clock proxy).
  • 두 측정이 0.5%p 안에 일치 — 측정 방식이 다른데도 일치. 우연이 아닌 구조적 패턴.
  • "GPU가 외부 도구(Wikipedia API) 응답을 기다리는 시간"이 단일 최대 비효율 — decode-RAG가 직접 노릴 수 있는 영역.

Fig 7 — 단대단 지연 분포 (Heavy Tail)

해설 (4 agent 분포 enriched)
  • 일치 패턴: ReAct 지연이 heavy tail. ReAct median 27.7s, p95 87.2s, max 115s, p95/p50 = 3.15 (검증 기준 ≥ 2.0 PASS).
  • Reflexion 분포 (n=50, 노랑 선): median 73.5s, p95 234s, p95/p50 = 3.19 — ReAct와 동일 ratio지만 분포가 우측으로 평행 이동 (반성 단계가 절대값을 키움).
  • LLMCompiler 가장 heavy tail (n=50, 회색 선): p95/p50 = 7.63 — bimodal 분포 (~5s 실패 클러스터 + ~90s 시도 클러스터). executor 깨짐의 직접 증거.
  • 절대값(87.2s vs 논문 20.7s)이 4-5배 큰 이유: M3 Pro Q4가 A100 + vLLM보다 단위 처리 속도 느림.
  • 시사점 — agentic workload는 평균이 아닌 p95를 봐야 함. mean 기반 SLA는 실제 사용자가 보는 latency를 과소평가.

Fig 8 — 입력/출력 토큰 분해

해설 (논문 색상 매핑 + L4 한계 표시)
  • 총량 트렌드 일치: LATS input 1.3M 토큰 (회색 system 압도적) — 트리 모든 가지가 system prompt 재포함. ReAct 920K, Reflexion 620K, LLMCompiler 50K (executor 깨짐).
  • 색상 매핑 — 회색 = Instruction+Few-shot (논문 light/dark gray), 검정 = User, 녹색 = LLM history, 노랑 = Tool history, 빨강 = Output.
  • L4 한계 — input/tool = 0 across all agents: HotpotQA tool 응답이 LangChain HumanMessage로 래핑되어 우리 핸들러가 "tool" 카테고리로 못 잡음. 차트의 노랑 막대가 비어있는 이유. 페이퍼 6-way 분해(Instruction/Few-shot/User/LLM hist/Tool hist/Output) 대비 우리 4-way (system/human/ai/tool) 한계.
  • LLMCompiler 빈 막대: planner 1번만 돌고 executor 깨짐 → tokens_input/output = 0 (callback 미캡처). Fig 4의 "?"와 같은 원인.

Fig 13 — Accuracy vs Latency Pareto Frontier 핵심

해설 (논문 마커 매핑 + 한계 주석)
  • 마커 매핑: 빨강 사각 = ReAct, 노랑 동그라미 = Reflexion, 파랑 삼각 = LATS, 회색 다이아 = LLMCompiler — 모두 논문 Fig 13 convention과 동일.
  • 일치 핵심: LATS 가장 정확 + 가장 비싼 페이퍼 형상 재현 (33% / 1037s vs ReAct 32% / 41s → 25× 비용 / 1% 정확도 이득). LLMCompiler가 가장 빠름(27.8s)도 일치.
  • 차이: 우리 Reflexion(16%)이 ReAct(32%)보다 정확도 낮음 — 페이퍼와 반대. 원인: Q4 양자화에서 reflection 단계가 자기수정보다 노이즈 추가.
  • LLMCompiler 18% 주의: executor 깨져서 50쿼리 중 41개가 빈 답 → "정확도 18%"는 partial 결과 (차트에서 빨강 주석으로 표시).
  • LATS 통계 한계: 표본 3개로 Spearman ρ(latency, accuracy) = 0.40 (기준 ≥0.6 FAIL). 차트에서 회색 주석으로 표시.
PART 4
Serving · 캐싱 · Test-Time Scaling
우리가 미실행한 영역 — 논문 결과 중심으로 정리

Fig 9 — Prefix Caching 효과 논문 only

해설
  • 에이전트는 반복 호출마다 input prefix를 공유 → prefix caching으로 prefill 평균 60.1% 감소.
  • 단대단 지연 평균 15.7% 감소. 모듈 단위 개선이 시스템 전체에 큰 영향.
  • CoT는 호출 1회뿐이라 효과 없음. 에이전트 워크로드에서만 효과 큼.
  • 우리 한계: fig13_pareto_cache_on.yaml sweep 설정은 있으나 시간 제약으로 실행 못 함. 패턴 재현은 향후 작업.

Fig 10 — AI Agent Serving 시스템 아키텍처 논문 only

해설
  • 다중 사용자 동시 요청을 처리하는 serving 시스템 개요.
  • 각 사용자 요청은 독립적인 agent loop를 가짐 → 자원 스케줄링 복잡도 증가.
  • 단일 요청 내(intra-request)에서는 LLM-tool 순차 의존성으로 병렬화 어려움 → inter-request 병렬성으로 자원 활용 끌어올려야 함.

Fig 11 / 12 — 95%-ile latency vs QPS, KV cache 메모리 논문 only

해설
  • Fig 11: ChatGPT vs 에이전트 비교 QPS-throughput 곡선. 에이전트는 같은 QPS에서 훨씬 높은 p95 — prefix caching이 곡선을 우측 평탄화.
  • Fig 12: prefix caching으로 KV cache 평균 사용량 51.7% 감소, 최대 63.5% 감소. GPU 메모리 자원 효율 활용.

Fig 14 / 15 — Iteration · Few-shot Scaling (ReAct) 논문 only

해설
  • Fig 14 (Iteration): 평균 지연·정확도는 빠르게 포화. 그러나 p95 지연은 계속 선형 증가 — outlier가 budget을 모두 소진해 latency tail만 키움.
  • Fig 15 (Few-shot): 예시 추가는 초기 정확도 향상. 흥미롭게도 평균 지연은 오히려 감소 — 좋은 예시가 더 적은 reasoning 단계로 풀게 도와줌.
  • 파란 다이아몬드 = accuracy/latency 비율 최대 = 가장 비용-효과적 튜닝 포인트.
  • 우리 한계: fig14_iteration.yaml, fig15_fewshot.yaml sweep config 보유. 시간 제약으로 미실행.

Fig 16 — Test-Time Scaling: Sequential vs Parallel 논문 only

해설
  • (a) Reflexion sequential: reflection step 2 → 32. 정확도 ↑ 하지만 한계 비용 31× 증가 (16.9→25.6s에서 +4% / 56s→325s에서도 +4%).
  • (b) LATS sequential: 동일한 수확체감 패턴.
  • (c) LATS parallel: 자식 노드 1→16 늘리면 정확도 +14.4%p, 동시에 평균 지연 -196.3s. 단 동시 LLM 호출 폭증 → 자원 제약 환경에서 어려움.
  • 정책 시사: 지연 민감 워크로드는 parallel scaling, 자원 제약 환경은 sequential.
  • 우리 한계: fig16{a,b,c}.yaml sweep 보유 (fig16b는 iter=32 cut). 시간상 미실행.

알려진 한계 (재현성 제약)

하드웨어·모델 격차

  • L1 — A100 + vLLM (논문) → M3 Pro + llama.cpp (우리). 절대 throughput 약 5-10× 느림. 시간값 직접 비교 불가, 패턴만 비교.
  • L2 — Llama-3.1-8B Q4_K_M 양자화 (논문은 FP16). Q4 노이즈가 LATS(193 LLM calls)에 가장 크게 누적. Reflexion이 ReAct보다 정확도 낮은 원인 후보.
  • L6 — NVIDIA DCGM 없음. Apple GPU 등가물 없어 wall-clock proxy 사용. 의외로 54.5% vs 54.98%로 일치했지만 의미가 정확히 동일하지는 않음.

표본·인프라 한계

  • L3 — LATS 표본 3개 (논문 50). M3 Pro에서 LATS 1쿼리당 17-40분 소요. Fig 13 Spearman FAIL의 직접 원인.
  • L4 — Reflexion·LATS·LLMCompiler의 tool 호출이 LangChain BaseTool.invoke를 안 거쳐 우리 핸들러가 못 잡음. Tool 카운트는 사후 구조적 복구(recover_tool_calls.py) — Reflexion ~14, LATS ~55. Tool 실행 시간(tool_ms)과 토큰 분해는 복구 불가 — Fig 8 FAIL의 원인.
  • L5 — Wikipedia API live 호출. 논문(2026-01)과 우리(2026-05) 사이 4개월간 내용 변동 가능성.
  • L7 — Serving 실험(Fig 10/11/12) 미수행. decode-RAG는 single-request부터 검증 가능하므로 영향 적음.

결론

패턴 수준에서 핵심 메시지 재현 성공

M3 Pro + Llama-3.1-8B Q4 환경에서도 페이퍼의 "동적 추론의 인프라 비용" 핵심 메시지가 동일한 형상으로 재현됨. 절대 수치가 아닌 비율·분포·트레이드오프 패턴이 일치.

재현된 핵심 패턴 5가지

  • 🎯 GPU idle 54.5% → 54.98% — 단 0.48%p 차이 (논문 NVIDIA DCGM, 우리 wall-clock proxy)
  • 📈 LATS LLM 호출 폭증 — 논문 71×, 우리 22× (같은 자릿수 트렌드)
  • ⏱ Tool wait가 latency 지배 — ReAct HotpotQA에서 단일 최대 컴포넌트 (우리 44%)
  • 📊 Heavy-tail 분포 — p95/p50 = 3.34 (논문도 ShareGPT 대비 훨씬 wider)
  • 💰 LATS 수확체감 — 25× wall-clock 비용 / 1% accuracy 이득 (논문 31× 패턴과 형상 일치)
decode-RAG 연구에의 함의 — ReAct GPU idle 55%, Tool wait 44%, p95/p50 = 3.34는 모두 decode 중 retrieval prefetch/overlap 기법이 직접 노릴 수 있는 비효율 영역. 본 재현이 그 motivation의 정량적 baseline을 제공.
