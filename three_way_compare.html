decode-RAG · 3-WAY COMPARISON

같은 분석, 세 출처 — 논문 vs 1차 vs 현재

동일 지표를 논문(원본 그래프) · 1차 실험(Live Wikipedia ReAct) · 현재 실험(Cohere 벡터DB ReAct) 세 출처로 나란히 비교. 환경이 다르므로 절대 시간이 아니라 비율·구성·분포 형상을 본다.

논문 arXiv 2506.04301v2 (HPCA-2026, KAIST) 워크로드 HotpotQA 1차·현재 Llama-3.1-8B Q4 · M3 Pro 표본 1차 50 · 현재 139문항

시작하기 전에 — 초보자용 안내

📖 한 문장 요약 AI 챗봇(LLM)에게 '검색 도구'를 쥐여주고 스스로 자료를 찾아가며 답하게 만든 게 AI 에이전트다. 그런데 검색 결과를 기다리는 동안 비싼 GPU가 절반 가까이 논다(idle). 이 노는 시간을 측정하고, 검색을 미리 당겨와 숨겨서 줄이려는 것이 이 연구의 목표인 decode-RAG다. 이 보고서는 그 출발점으로, 검색 방식을 바꿔가며 "검색이 전체 시간의 몇 %를 먹는지"를 논문·1차·현재 세 출처로 비교한다.
핵심 용어 풀이 — 클릭해서 펼치기/접기
① 큰 그림
LLM (거대 언어 모델)ChatGPT 같은 'AI 두뇌'. 이 실험은 작은 오픈 모델 Llama-3.1-8B를 노트북급 Mac(M3 Pro)에서 돌렸다.
AI 에이전트 (Agent)LLM에 검색 같은 '도구'를 붙여, 한 번에 답하지 않고 생각→검색→관찰을 반복하며 스스로 답을 찾는 시스템.
ReAct가장 기본적인 에이전트 방식. Reason(생각)+Act(행동) — 생각하고→검색하고→결과를 보고→다시 생각하기를 반복한다.
RAG (검색 증강 생성)LLM이 답하기 전에 외부 문서를 검색(retrieval)해 참고하는 방식. 모르는 건 찾아보고 답한다.
decode-RAG (이 연구의 목표)LLM이 답을 한 토큰씩 생성(decode)하는 동안, 다음 검색을 미리 당겨와 겹쳐서 '기다리는 시간'을 숨기는 기법.
② 시간 · 하드웨어
e2e latency (전체 지연)질문을 받고 최종 답까지 걸린 총 시간(초). 'end-to-end'의 약자. 이 보고서 비율 계산의 분모.
GPU / GPU idleGPU는 LLM 계산을 하는 비싼 장치. idle = GPU가 아무것도 안 하고 노는 시간(주로 검색을 기다릴 때 생김).
prefill / decodeLLM이 일하는 두 단계. prefill=받은 글을 한꺼번에 빠르게 읽기. decode=답을 한 토큰씩 순차 생성(메모리 대역폭에 묶여 느림).
token (토큰)LLM이 다루는 텍스트의 최소 단위. 대략 단어 하나 또는 단어 조각.
③ 검색 (retrieval)
retrieval / tool (검색 도구)질문에 필요한 문서를 찾아오는 동작. 이 보고서에서 'tool'은 곧 검색을 뜻한다.
Live Wikipedia (1차)실시간 위키피디아 키워드 검색. 느리고(~2초/회) 결과가 시간에 따라 바뀌어 재현이 어렵다.
Cohere 벡터DB (현재)문장을 숫자 벡터로 바꿔(embedding) 의미가 비슷한 문서를 빠르게 찾는 방식. FAISS로 88만 문서에서 ~0.4초에 검색, 고정본이라 재현 가능.
hop / multi-hop답에 도달하기까지 거치는 검색 단계 수. multi-hop = 여러 문서를 거쳐야 답이 나오는 어려운 질문.
HotpotQA실험에 쓴 질문 데이터셋(총 7,405문제, 이번엔 앞 50개 사용). 대부분 multi-hop 질문이다.
dense / 의미 기반 검색단어가 정확히 겹치지 않아도 '뜻'이 비슷하면 찾아주는 검색. 키워드 검색보다 똑똑하다.
④ 통계 용어
EM accuracy (정확도)Exact Match. 모델 답을 정답(gold)과 정규화(소문자·관사/구두점 제거) 후 완전히 같으면 정답으로 친 비율. 부분 점수 없음('15 August 1843' ≠ '1843').
F1 score (부분 점수)답을 토큰(단어)으로 쪼개 정답과 겹치는 정도를 0~1로 잰다. '거의 맞은' 답에 부분 점수('15 August 1843' vs '1843' → F1 0.5). HotpotQA 공식 지표는 EM·F1 둘 다.
median(중앙값) · p50 · p95median=p50=크기순 한가운데 값(평균보다 이상치에 강함). p95=느린 쪽 상위 5% 지점. 분포를 볼 때 쓴다.
heavy tail (두꺼운 꼬리)대부분 빠른데 소수가 아주 느려서, 분포 그래프의 오른쪽이 길게 늘어진 모양.
share / fraction (비율)전체를 100%로 놓았을 때 한 부분이 차지하는 몫(%). 이 보고서의 핵심은 'tool이 e2e에서 차지하는 비율'.
⑤ 이 보고서의 세 출처
📄 논문재현 대상인 KAIST 논문 "The Cost of Dynamic Reasoning"(HPCA-2026). 고성능 A100 GPU·vLLM 환경에서 측정.
📊 1차기존 Live Wikipedia 검색으로 측정한 우리 실험(Llama-3.1-8B, M3 Pro).
📊 현재Cohere 벡터DB + 로컬 cross-encoder rerank(FAISS top-50 → bge-reranker-base → top-5)로 검색한 우리 실험. 1차와 LLM·환경은 같다. 표본 139문항(전체 7,405 스케일업 중 Cohere 쿼터로 일시 중단).

💡 아래 각 지표(①~⑥)마다 '자세한 해석' 블록이 있어, 이 용어들이 실제 그래프에서 어떻게 쓰이는지 풀어 설명한다.

⚠️ 비교의 전제 — 반드시 같이 읽을 것 논문은 A100 · vLLM · FP16, 우리는 M3 Pro · llama.cpp · Q4다. 절대 시간(초·Wh)은 비교 불가 — 비율·구성·분포 형상·순위만 비교한다. 또한 지표 ①②⑥의 논문 수치는 5종 에이전트·4벤치 평균이라 ReAct-HotpotQA 단독값이 아니다 (⑤만 논문이 ReAct-HotpotQA를 직접 제공). 1차는 HotpotQA 전체 7,405문항 중 앞 50문항(0.68%), 현재는 앞 139문항(1.9%) 표본 기반이다. 현재 = Cohere 벡터DB + rerank(이전 dense-only 50문항 baseline을 대체).
📐 연구 방법 — 정확도는 EM과 F1 둘 다로 잰다 HotpotQA 공식 평가 지표인 EM(Exact Match, 완전 일치)과 F1(토큰 단위 부분 점수)을 함께 쓴다. EM은 엄격해 '거의 맞은' 답도 0점 처리하므로(예: '15 August 1843' vs 정답 '1843'), 부분 점수를 주는 F1을 같이 봐서 실제 성능을 보완 측정한다. 채점기는 논문 표준 hotpot_evaluate_v1.py와 동일하게 맞췄다(소문자·관사/구두점 정규화 후 비교). EM은 그대로 두고 F1을 추가했다.
⚖️ 누가 채점하나 — 사람도 LLM 심판도 아닌 결정론적 함수 EM·F1 둘 다 사람이나 다른 AI(LLM-judge)가 아니라 순수 파이썬 함수가 기계적으로 채점한다. 평가에 들어가는 건 딱 셋 — 에이전트의 최종 답(predicted), HotpotQA 데이터셋에 사람이 미리 적어둔 정답(gold), 그리고 채점 함수다. 즉 '기준' = gold 정답, '채점자' = 문자열 비교 함수이고, 같은 입력이면 항상 같은 점수가 나오는 결정론적(deterministic) 방식이라 재현 가능하다. 이 함수는 임의로 만든 게 아니라 논문/벤치마크 표준 hotpot_evaluate_v1.py(AgentBench hotpot_evaluate.py)를 byte-for-byte 그대로 포팅해 (repro/measurement/eval.py), 우리 EM·F1이 논문 채점기와 정확히 일치한다. EM은 정규화 후 완전 일치면 1, F1은 같은 정규화를 거친 답을 토큰으로 쪼개 겹침을 2·P·R/(P+R)로 0~1 점수화한다(yes/no/noanswer는 F1도 완전 일치만 인정).

지표별 3자 비교

① Retrieval-time fraction (tool ÷ e2e) — 헤드라인

논문 GPU idle 54.5%(HotpotQA, retrieval 대기) · tool share 30.2%(평균)  ·  1차 44.8% (median 46.5%)  ·  현재 31.3% (median 31.1%). 현재는 빠른 dense 검색에 rerank를 더해 retrieval를 정당하게 무겁게 만든 결과 — dense-only일 땐 11%로 떨어졌으나, rerank로 ~31%가 되어 논문 tool share 30.2%·1차 45%대와 같은 자릿수로 돌아왔고 재현 가능하다. ⚠️ 환경이 달라 절대값 비교 불가 — 형상·비율만. 논문 30.2%는 에이전트·벤치 평균. 현재 n=139.

자세한 해석 — 무슨 뜻이고, 왜 세 값이 다른가

이 지표가 측정하는 것

tool ÷ e2e = 한 질문을 푸는 전체 시간 중 검색 응답을 기다린 시간의 비율. ReAct는 생각 → 검색 → 관찰을 반복하는데, 검색을 기다리는 동안 LLM(GPU)은 논다(idle). 이 노는 비율이 곧 decode-RAG가 숨길 수 있는 비효율의 상한이다.

세 그래프 읽는 법

  • 📄 논문 Fig 6 — GPU 시간 분해 막대(🔴prefill + 🩷decode + ⬛idle). HotpotQA의 ReAct 칸에서 검정(idle) ≈ 54.5%. 반면 CoT(도구 없음)는 idle이 거의 0 → "도구 쓰는 에이전트가 GPU를 놀린다"가 논문의 메시지.
  • 📊 1차 · 현재 — 점 하나 = 질문 하나(각 50개), 박스 = 가운데 50%, 굵은 선 = median. 1차는 30~60%로 넓게 퍼지고(med 46.5%), 현재(rerank)는 20~40%대에 몰린다(med 31.1%).

tool 총시간 = 검색당 시간 × 검색 횟수. 백엔드를 바꿔가며 이 둘이 어떻게 변하는지:

dense-only는 검색이 너무 빨라(~415ms) 비율이 11%로 급락했다 — "RAG가 너무 빠르다"(⑦). 그래서 rerank(FAISS top-50 → cross-encoder → top-5)를 더해 검색을 정당하게(인위적 지연이 아니라 실제 연산으로) ~1,946ms로 무겁게 했다. 그 결과 tool이 1.2s→6.3s로 커지고 비율이 11% → 31%로 올라 논문 tool share 30%·1차 45%대와 같은 자릿수로 돌아왔다. 핵심: 검색을 의미있게 무겁게 만들어야 decode-RAG로 숨길 여지가 생긴다.

논문 54.5% ≈ 1차 44.8% — 비슷하면서 왜 다른가

둘 다 느린 라이브 검색이라 같은 ~45–55%대(패턴 일치). 미세 차이는 측정 방식(논문 = GPU idle, DCGM 하드웨어 측정 / 1차 = tool÷e2e wall-clock)과 LLM 속도(A100/vLLM은 LLM이 빨라 검색 대기 비중↑, M3/Q4는 LLM이 느려 분모 e2e가 커지며 비중↓) 때문.

decode-RAG 연구에의 의미

1차 ~45% = 상한(비재현, 라이브 API 드리프트), 현재 ~11% = 재현 가능한 하한(로컬 FAISS만이면 ~2%). 두 끝점으로 bracket하면 앞으로 만들 prefetch/overlap 기법의 잠재 이득 범위를 정량화할 수 있다. 역설: 좋은 검색일수록 이 비율이 낮아진다 → decode-RAG의 동기는 검색이 느리거나 코퍼스가 큰 환경에서 더 강해진다.

⚠️ 논문 막대(GPU idle, DCGM)와 우리 box(tool/e2e, wall-clock)는 정의가 달라 "같은 축 정밀 비교"가 아니라 형상·자릿수 비교다. 현재는 139문항, 1차는 50문항 표본 기반.

② Latency 구성 (LLM / tool / overhead)

논문 LLM 69.4% / tool 30.2% (평균)  ·  1차 LLM 55% / tool 45%  ·  현재 LLM 68% / tool 31%. rerank로 tool 조각이 다시 커져(11%→31%) 현재 68/31이 논문 69/30과 거의 일치한다. ⚠️ 논문값은 5종 에이전트·4벤치 평균. 우리는 ReAct-HotpotQA. 현재 n=139.

자세한 해석 — Y축이 뭐고, 이 그래프가 의미하는 것

Y축 = "share of e2e latency (%)" — 전체 시간의 100% 분해

한 질문의 전체 처리 시간(e2e)을 100%로 놓고, 그 시간이 어디에 쓰였는지 쌓아올린(stacked) 막대다. 🔴LLM(생각·추론 계산) + ⬛Tool(검색 대기) + 회색 overhead가 합쳐서 100%. ①이 'tool 조각 하나'만 따로 본 것이라면, ②는 그 tool + 나머지(LLM)까지 전부 보여준다. 그래서 ②의 tool% = ①의 평균 비율과 같다(1차 45%≈44.8%, 현재 31%≈31.3%).

막대 읽는 법

  • 아래 🔴빨강 = LLM 계산(prefill+decode), 위 ⬛검정 = Tool(검색 대기), 맨 위 회색 = overhead(거의 0이라 안 보임).
  • 조각 높이 = 그 비율. 합은 항상 100%라서 한쪽이 커지면 다른 쪽이 자동으로 작아진다(zero-sum).

세 그래프가 말하는 것

  • 📄 논문 Fig 5 — 평균 LLM 69.4% / tool 30.2%. 분홍(LLM+Tool overlap)은 LLMCompiler가 도구를 LLM과 겹쳐 비동기 실행한 부분 — 이게 바로 decode-RAG가 노리는 "검색 숨기기"의 선례다.
  • 📊 1차 LLM 55% / tool 45% — 느린 라이브 검색이 거의 절반을 먹는다.
  • 📊 현재 LLM 68% / tool 31% — rerank로 tool 조각이 다시 커져 논문(69/30)과 거의 일치한다.

왜 현재가 LLM 68% / tool 31%인가 (dense-only 88/11에서 회복)

dense-only일 땐 검색이 ~5배 빨라(2,004→415ms) tool이 11%로 쪼그라들고 LLM이 88%를 먹었다. 여기에 rerank(cross-encoder 연산)를 더해 검색을 ~1,946ms로 무겁게 하니 tool 조각이 31%로 커졌다. 전체가 100%라 tool이 커진 만큼 LLM 비중은 88%→68%로 내려와 논문 69/30 구성과 거의 일치한다.

decode-RAG에의 의미

검정(tool) 조각이 클수록 = LLM이 노는 시간이 크다 = decode-RAG로 숨길 여지가 크다. dense-only(11%)는 여지가 거의 없었지만, rerank로 31%까지 키워 숨길 여지를 의미있게 확보했다. 이게 ⑦에서 hop별 retrieval≈decode(1:1)를 맞춘 것과 같은 맥락이다 — decode-RAG 기법을 실험할 토대가 마련됐다.

⚠️ 논문값은 5종 에이전트·4벤치 평균이라 ReAct-HotpotQA 단독값이 아님. overhead는 거의 0. 현재 n=139, 1차 n=50 표본.

③ Calls per query (LLM · tool)

논문 도구 에이전트는 CoT 대비 LLM 호출 9.2×(LATS 71×)  ·  1차 LLM 8.84 / tool 8.54  ·  현재 5.67 / 5.50. 커버리지 높은 dense+rerank 검색이 더 적은 hop으로 정답에 도달(8.5→5.5회). ⚠️ 논문은 ReAct 절대 호출수를 표기하지 않음(비율만). ReAct는 구조상 LLM≈tool 호출. 현재 n=139.

자세한 해석 — Y축이 뭐고, 이 그래프가 의미하는 것

Y축 = "calls per query (mean)" — 질문당 평균 호출 횟수

한 질문을 푸는 데 LLM을 몇 번 부르고(왼쪽 막대), 검색 도구를 몇 번 호출했는지(오른쪽 빗금 막대)의 평균. ReAct는 생각(LLM 호출) → 검색(tool 호출) → 관찰을 반복하므로 구조상 LLM 호출 ≈ tool 호출이다.

세 그래프가 말하는 것

  • 📄 논문 Fig 4 — 도구 에이전트는 CoT(1회) 대비 LLM 호출이 평균 9.2배, LATS는 71배까지 폭증. "동적 추론이 LLM 호출을 폭증시킨다"가 메시지. (ReAct 단독 절대 횟수는 표기 안 함 — 비율만.)
  • 📊 1차 LLM 8.84 / tool 8.54회 — 라이브 검색이 부정확해 더 많은 hop을 돈다.
  • 📊 현재 LLM 5.67 / tool 5.50회 — dense+rerank 검색이 정확해 1차(8.5회)보다 적은 횟수로 정답에 도달.

왜 8.5회 → 5.5회로 줄었나

Cohere 벡터DB는 의미 기반(dense) 검색이고 rerank로 관련도가 더 올라, 한 번에 더 관련된 문서를 가져온다. "다시 검색"이 줄어 hop 수가 감소한다(어려운 multi-hop 질문만 많이 돈다). 호출이 적다는 건 왕복(round-trip)이 적다 = 그만큼 idle도 적다는 뜻.

decode-RAG에의 의미

호출 횟수 = LLM↔검색 왕복 횟수 = decode-RAG가 끼어들(겹칠) 지점의 개수. 횟수가 많을수록(1차) 숨길 기회가 많지만 그만큼 비효율도 크다. 적은 호출(현재)은 효율적이지만 prefetch로 숨길 여지는 작아진다.

⚠️ 논문은 ReAct 절대 호출수 대신 비율(9.2×)만 제시. 현재 n=139, 1차 n=50 표본.

④ 정확도 — EM(엄격) + F1(부분 점수)

논문 Fig 13 산점도(ReAct는 표로 미기재)  ·  1차 EM 32.0% / F1 38.5%  ·  현재 EM 30.9% / F1 41.9%. 현재가 F1 +3.4%p(부분 점수 우위, "거의 맞은" 답이 더 많음)지만 EM은 −1.1%p로 비슷 — dense+rerank가 증거 질을 높여도 EM은 표본·난이도에 민감. ⚠️ 논문 Table III는 Reflexion/LATS만 수록. 표본 불일치(현재 n=139 vs 1차 n=50)라 직접 비교 아님 — 같은 첫 50문항 기준일 땐 거의 불변.

자세한 해석 — Y축이 뭐고, 이 그래프가 의미하는 것

그래프의 두 막대 = EM(엄격) + F1(부분 점수)

EM (Exact Match) = 모델 답이 정답과 정규화(소문자·관사/구두점 제거) 후 완전히 같아야 1점. 부분 점수가 없어 '거의 맞은' 답도 0점이다(예: '15 August 1843' vs '1843'). F1 = 답을 토큰으로 쪼개 겹치는 정도를 0~1로 주는 부분 점수(그 예의 F1은 0.5). HotpotQA 공식 지표가 이 둘이라 함께 본다.

세 그래프가 말하는 것

  • 📄 논문 Fig 13 — Accuracy-vs-Latency 산점도. ReAct-HotpotQA의 정확도는 표로 적혀있지 않고 산점도 점으로만 표시(논문 Table III는 Reflexion/LATS만 수록).
  • 📊 1차 EM 32.0%(16/50) / F1 38.5%  ·  📊 현재 EM 30.9% / F1 41.9% (n=139).

현재(dense+rerank)는 F1이 1차보다 높다 (F1 +3.4%p, EM −1.1%p)

키워드 기반 라이브 검색 → 의미 기반(dense)+rerank로 바꾸니 더 관련 있는 문서를 가져와 증거(context)의 질이 올라간다. 그 결과 F1(부분 점수)은 +3.4%p 올랐다 — 완전히 맞히진 못해도 '정답에 더 가까운' 답이 늘었다. EM은 −1.1%p로 사실상 비슷한데, 표본이 다르다(현재 n=139 vs 1차 n=50)는 점을 감안해야 한다(같은 첫 50문항만 보면 거의 불변이었다).

decode-RAG에의 의미

핵심: rerank로 검색을 무겁게(느리게) 만들었는데도 정확도는 떨어지지 않고 F1은 오히려 올랐다. 즉 retrieval를 키워 decode-RAG로 숨길 여지를 확보하면서도 정확도 손해가 없다 — "비율 맞추기"가 공짜에 가깝다.

⚠️ 표본 불일치(현재 n=139 vs 1차 n=50)라 EM/F1 직접 비교는 주의. EM은 부분 점수가 없어 보수적이라 F1을 함께 본다. 논문 정확도는 산점도에서 읽은 값.

⑤ Latency 분포 — heavy tail (p95 / p50)

논문 ReAct-HotpotQA p95 = 20.7s (이 지표만 논문이 직접 제공)  ·  1차 p50 27.7s / p95 87.2s (3.15×)  ·  현재 p50 20.3s / p95 94.5s (4.66×). rerank가 검색을 무겁게 해 현재 p50이 20.3s로 올랐고(dense-only 10.5s 대비), 세 출처 모두 heavy-tail 형상을 재현. ⚠️ 절대 초는 환경차로 다름 — 꼬리의 형상/비율만 비교. 현재 n=139.

자세한 해석 — X·Y축이 뭐고, 이 그래프가 의미하는 것

X·Y축 = 지연 시간의 분포

X축 = 한 질문의 e2e 지연(초), Y축 = 그 지연이 나온 빈도 밀도(histogram). 즉 "몇 초짜리 질문이 얼마나 많은가". 점선은 p50(중앙값, 검정)과 p95(상위 5%, 빨강).

heavy tail이란 — p95/p50 비율

대부분 질문은 빠른데(p50) 소수의 어려운 multi-hop 질문이 아주 느려서 오른쪽으로 긴 꼬리를 만든다. p95/p50가 클수록 꼬리가 두껍다(편차가 크다). 논문도 같은 heavy-tail을 보고함 — 정적 챗봇(ShareGPT)엔 없는 에이전트 특유의 현상.

세 그래프가 말하는 것

  • 📄 논문 Fig 7 — ReAct-HotpotQA p95 = 20.7s (이 지표만 논문이 ReAct-HotpotQA를 직접 제공).
  • 📊 1차 p50 27.7s / p95 87.2s → 3.15×
  • 📊 현재 p50 20.3s / p95 94.5s → 4.66×

현재의 꼬리 비율(4.66)은 1차(3.15)보다 크다

현재는 rerank로 검색이 무거워져 p50이 20.3s(dense-only 10.5s보다 느림). 쉬운 질문은 적당히 느려지지만 어려운 multi-hop 질문은 검색을 많이 돌아 p95가 94.5s까지 늘어, 빠른 질문과 느린 질문의 격차가 벌어져 꼬리 비율이 커진다. 세 출처 모두 같은 heavy-tail 형상을 재현하는 것이 핵심.

decode-RAG에의 의미

꼬리(느린 질문)일수록 검색 왕복이 많아 LLM이 더 오래 논다 = decode-RAG 이득이 꼬리에서 가장 크다. 평균이 아니라 p95 같은 꼬리를 줄이는 것이 서빙(throughput) 관점에서 특히 중요하다.

⚠️ 절대 초는 환경(A100 vs M3)차로 직접 비교 불가 — 꼬리의 형상/비율만. 현재 n=139, 1차 n=50 표본.

⑥ Prefill / decode 분할 (추정)

논문 active LLM 시간 중 prefill 4.7% / decode 74.1%  ·  1차 prefill 26% / decode 74%  ·  현재 prefill 39% / decode 61%. 세 출처 모두 decode 우세 형상이 일치. ⚠️ 우리값은 wall-clock 추정치(*_estimate)이고 분모가 다름(논문=idle 포함 GPU 시간, 우리=active-LLM 시간) — 형상만.

자세한 해석 — Y축이 뭐고, 이 그래프가 의미하는 것

Y축 = "share of active LLM time (%)" — LLM 계산 시간의 분해

②가 e2e 전체를 LLM/tool로 나눴다면, ⑥은 그중 LLM이 실제로 계산하는 시간만 떼어 prefill(입력 프롬프트를 한 번에 읽어들이는 단계)과 decode(답 토큰을 하나씩 생성하는 단계)로 나눈 것. 합쳐서 100%.

prefill vs decode — 왜 중요한가

prefill은 입력을 한꺼번에 처리해 빠르고, decode는 답을 한 토큰씩 순차 생성하는데 연산량보다 메모리 대역폭에 묶여(memory-bound) 느리다(논문 §V-A). decode 우세면 "답을 생성하느라 시간을 쓴다"는 뜻. prefix caching 같은 최적화는 주로 prefill을 줄인다.

세 그래프가 말하는 것

  • 📄 논문 Fig 6 — active LLM 시간 중 prefill 4.7% / decode 74.1%(나머지는 idle). decode 압도적.
  • 📊 1차 prefill 26% / decode 74%  ·  📊 현재 prefill 39% / decode 61%.

세 출처 모두 decode가 우세한 같은 형상 → 패턴 일치. 현재가 prefill 비중이 좀 더 큰 건 rerank로 가져온 검색 결과(context)를 매번 다시 읽어들이는 prefill 부담이 상대적으로 커서다.

decode-RAG에의 의미 + 주의

decode가 우세하다는 건 decode 도중에 검색을 미리 당겨올(prefetch) 시간 여유가 충분하다는 뜻 — decode-RAG의 전제가 성립함을 보여준다. 다만 이 값은 정밀 측정이 아니라 추정치다.

⚠️ 우리 prefill/decode는 wall-clock *_estimate(추정치)이고 논문은 분모가 다름(논문=idle 포함 GPU 시간, 우리=active-LLM 시간). 절대값이 아니라 "decode 우세" 형상만 비교. 현재 n=139, 1차 n=50 표본.

⑦ 심화 — 현재 백엔드의 hop별 retrieval·decode 균형

📌 이 섹션은 3자 비교가 아님 위 ①~⑥은 논문·1차·현재 3자 비교였다. ⑦은 현재(Cohere 벡터DB) 백엔드 내부를 hop 단위로 쪼개, "각 검색이 그 직후 LLM 작업(decode)에 비해 얼마나 짧은가"를 본다 — decode-RAG로 검색을 숨길 여지가 hop마다 얼마나 되는지 측정하는 출발점이다.

한 hop = 검색 1회 + 그 직후 LLM call

기존 react_vectordb 50개 trace에서 198개 hop을 페어링(검색 span → 직후 LLM span). retrieval은 wall-clock(정확), decode는 두 가지로 측정: LLM 전체(prefill+decode)와 순수 decode(추정치, 신뢰 hop 152개).

거의 모든 hop이 1:1 선 아래 — retrieval이 decode보다 훨씬 짧다. median 비율 0.12(LLM 전체 기준) / 0.23(순수 decode 기준). 즉 1:1로 맞추려면 retrieval을 약 8.6배(LLM 전체) ~ 4.3배(decode) 키워야 한다. ⚠️ 지금은 retrieval이 작아 decode-RAG로 숨겨도 이득이 작다 — 이게 "RAG가 너무 빠르다"의 정량.

자세한 해석 — 그래서 어떻게 1:1에 맞추나

왜 hop 단위로 보나

decode-RAG는 한 hop의 검색 대기를 그 다음 LLM이 답을 생성하는 시간 뒤로 숨긴다. 따라서 hop마다 retrieval ≈ decode여야 검색을 거의 다 가릴 수 있다(이득 최대). retrieval이 너무 짧으면 숨겨도 미미.

현재 측정 결과

  • retrieval median 438ms · decode-only median 1,967ms · LLM-total median 4,006ms
  • 비율 median: retrieval/decode-only 0.23, retrieval/LLM-total 0.12
  • 대부분 hop이 1:1 선 한참 아래(분포가 0~0.4에 몰림). 소수 outlier(느린 검색, 비율 1~5)만 위쪽.

1:1에 맞추는 정당한 레버 (다음 단계)

인위적 지연(sleep)은 연구적으로 무의미하므로 실제 연산으로 retrieval을 키운다: ① reranking 추가(cross-encoder/Cohere rerank, hop당 +50~200ms) ② top_k 확대(5→50) ③ 코퍼스/인덱스 확대 ④ 무거운 임베딩. 현재 retrieval 438ms의 대부분이 Cohere API 네트워크라, 진짜 "검색 연산"을 늘리는 rerank가 가장 자연스럽다.

주의 — decode 기준에 따라 목표가 다름

순수 decode만 겹친다면 ~4.3배, LLM 전체(prefill 포함)를 겹친다면 ~8.6배가 1:1 목표다. decode-RAG가 실제로 어느 구간을 겹칠 수 있는지(prefill 중에도 prefetch 가능한지)가 이 선택을 정한다.

⚠️ decode-only는 추정치(coarse_attribution 제외, 152/198 hop). retrieval·LLM-total은 wall-clock 전수. 분석 스크립트: repro/analysis/plot_hop_balance.py.
