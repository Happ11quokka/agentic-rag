PART 1논문 — 배경과 동기

왜 "에이전트의 인프라 비용"을 측정하는가, 그리고 그것이 decode-RAG와 어떻게 연결되는가.

The Cost of Dynamic Reasoning: Demystifying AI Agents and Test-Time Scaling from an AI Infrastructure Perspective (Kim et al., KAIST, arXiv 2506.04301v2, HPCA-2026)는 AI 에이전트(plan → tool → observe → refine를 반복하는 LLM 시스템)의 인프라 비용을 처음으로 정량 측정한 시스템 측정 연구다. 핵심 질문: "한 번에 답하는 정적 LLM 대비, 동적 추론 에이전트는 인프라 측면에서 얼마나 더 비싸고, 그만큼 정확도가 오르는가?"

왜 이 논문이 decode-RAG의 출발점인가 논문이 측정한 "GPU idle 54.5%"의 정체는 에이전트가 외부 도구(retrieval) 응답을 기다리며 GPU가 노는 시간이다. 즉 단일 요청 안에서 LLM↔도구의 순차 의존성 때문에 생기는 이 idle 구간이 바로 decode 중 retrieval를 미리 당겨와(prefetch/overlap) 숨길 수 있는 비효율이다. 본 연구의 측정 지표 retrieval-time fraction = tool_total / e2e 은 "그 idle를 얼마나 숨길 수 있는가"의 상한을 정량화한다.

PART 2기존 재현 완료

절대 수치(초·Wh)는 환경 격차로 비교 불가 → 비율·분포·순위·트레이드오프 형상의 재현을 목표로 함. "패턴 수준에서 일치하는가?"

  • GPU: NVIDIA A100 40GB ×1(8B) / ×8(70B)
  • Backend: vLLM 0.6.6 · FP16 · prefix cache on

  • 벤치마크: HotpotQA만 (decode-RAG에 가장 적합)

PART 3현재 실험 — ReAct × Cohere 벡터DB 완료

decode-RAG의 정량 baseline을 위해, 기존 라이브 Wikipedia 검색을 고정·재현 가능한 Cohere 밀집 벡터DB로 교체하고 retrieval-time fraction을 측정.

동기 — 왜 벡터DB baseline인가

기존 라이브 Wikipedia의 한계

  • 비재현성: 라이브 API는 콘텐츠가 시간에 따라 드리프트 (논문 2026-01 ↔ 우리 측정 사이 변동).
  • 네트워크 변동: 검색당 ~2s가 순수 네트워크 지연에 좌우 → 분리 측정 어려움.

Cohere 벡터DB가 주는 것

  • 고정 코퍼스: Cohere Wikipedia-2023-11 스냅샷 → 재현 가능.
  • 밀집 검색: 의미 기반 top-k. 키워드 검색보다 정확도↑.
  • 측정 가능한 비율: 검색이 표준 BaseTool.invoke 경로 → tool span 자동 캡처.

실험 설계

[오프라인 1회] Cohere en 스트리밍 ──filter(title ∈ HotpotQA)──▶ passages(text, title, emb) ├─▶ FAISS IndexFlatIP(1024) → cohere_hotpot.faiss (3.4 GB) └─▶ 정렬된 메타데이터 → cohere_hotpot_meta.parquet [런타임/sweep] ReAct ──search[q]──▶ Cohere embed(q) ──▶ FAISS top-k ──▶ top-k passages (LLM = 로컬 llama-server) (네트워크) (로컬) tool span = embed(API) + FAISS → tool_total_ms → decode-RAG 비율

인덱스: 880,777 passages · 60,682 titles · gold-title 커버리지 82.9% (11,421/13,783). 빌드 ~4.2h(415 샤드 스트리밍). 측정 경로는 기존 TraceCallbackHandler 재사용 — sweep 코드 무수정.

비교 그래프 — 라이브 Wikipedia vs Cohere 벡터DB

핵심 지표. 벡터DB는 검색이 빨라(415ms vs ~2s) 비율 자체는 44.8% → 11.1%로 낮아지지만, 분포가 좁고(5–29%) 재현 가능하다.

종합 비교. 벡터DB는 더 적은 검색(8.54→4.96)으로 더 정확하고(32→36%) 2.6배 빠르다(e2e median 27.7→10.5s).

실측으로 확인한 정정 — 커버리지가 오를수록 숨길 지연도 커진다 부분 빌드(40샤드, cov 7.2%)에선 비율 6.5%였으나 풀 빌드(415샤드, cov 82.9%)에선 ~10%로 상승. 이유: 커버리지↑ → 더 적은 hop으로 정답 도달(검색 10.7→5.0) → 대화 컨텍스트가 짧아져 LLM 호출당 시간↓, 동시에 인덱스가 커져 검색당 시간↑(305→415ms). 둘 다 retrieval 비중을 키운다. → 현실적 코퍼스일수록 retrieval fraction이 더 크다 = decode-RAG 동기에 유리한 방향.
decode-RAG 연구에의 함의
  • 이 실험은 재현 가능한 하한 baseline(~11%)을 제공한다. 라이브 Wikipedia(~45%)는 상한에 가깝지만 비재현적.
  • 두 백엔드를 양 끝점으로 두면 retrieval-time fraction의 범위를 bracket할 수 있어, 향후 prefetch/overlap 기법의 잠재 이득 한계를 정량화 가능.
  • 밀집 검색이 정확도까지 올린다는 점(+4%p)은, decode-RAG가 "속도와 정확도를 동시에" 노릴 여지를 시사.

PART 4종합 · 한계 · 향후 계획

지금까지의 진행

  • 논문 분석 — 5종 에이전트 인프라 비용 구조·핵심 figure 해독 완료.
  • 기존 재현 — Fig 4/5/6/7/8/13 재현. GPU idle 54.98%(Δ0.48%p), heavy tail p95/p50=3.34 등 패턴 일치.
  • decode-RAG baseline — ReAct×Cohere 벡터DB로 retrieval-time fraction 측정 (median 9.7%, 재현 가능). 라이브 Wikipedia(44.8%)와 bracket 확보.

알려진 한계

환경·모델 격차

  • L1·L2 — A100+vLLM+FP16(논문) ↔ M3 Pro+llama.cpp+Q4(우리). 절대 시간 직접 비교 불가, 패턴만.
  • L3 — LATS 표본 3개(쿼리당 17–40분) → Fig 13 Spearman FAIL.
  • L6 — NVIDIA DCGM 없음 → GPU idle은 wall-clock proxy.

측정·벡터DB 한계

  • L4 — Reflexion/LATS/LLMCompiler tool 호출이 콜백 우회 → tool 시간·토큰 분해 불가(Fig 8 FAIL).
  • 벡터DB-a — 쿼리 임베딩이 원격(Cohere API) → 완전 로컬 아님(의도된 선택: 비율 확보).
  • 벡터DB-b — ~11%는 재현 가능한 하한. 로컬 FAISS만이면 ~2%, 라이브 위키면 ~45%.

향후 계획 (다음 스텝)

  • decode-RAG 프로토타입 — decode 단계 중 retrieval를 prefetch/overlap하여 실제 e2e 지연 절감을 측정. baseline의 retrieval-time fraction이 곧 이론적 상한.
  • prefix caching sweep (Fig 9) — 캐싱 on/off로 prefill 절감 측정 → 캐싱과 decode-RAG 이득의 상호작용 분석.
  • 코퍼스 스케일 확장 — 풀 코퍼스(IVF-PQ)·로컬 dense 대조로 retrieval fraction의 상·하한을 더 넓게 bracket.
  • 일반화 — 70B/FP16 또는 다른 모델·다른 워크로드(WebShop 등)로 패턴 확인.
한 줄 요약 논문의 핵심 메시지(동적 추론의 인프라 비용, 특히 retrieval 대기로 인한 GPU idle)를 로컬에서 재현했고, 그 비효율을 재현 가능한 숫자(decode-RAG 비율 ~11% 하한 ↔ ~45% 상한)로 고정했다. 다음은 이 slack을 실제로 회수하는 decode-RAG 기법의 구현·측정.
